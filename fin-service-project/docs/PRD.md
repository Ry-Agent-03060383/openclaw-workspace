## 1. 项目概述

### 1.1 项目背景

随着金融科技的快速发展，传统金融服务模式面临数字化转型压力。中小企业融资难、风控效率低、数据孤岛等问题日益突出。本项目旨在构建一站式智慧金融服务平台，通过整合企业信用数据、自动化贷款审批流程、智能风险评估和可视化数据分析，提升金融服务效率和用户体验。

### 1.2 项目目标

- **业务目标**: 实现企业信用查询、贷款申请、风险评估全流程线上化
- **技术目标**: 构建高可用、可扩展的微服务架构
- **用户目标**: 为企业客户提供便捷的融资服务，为金融机构提供高效的 风控工具

### 1.3 目标用户

| 用户类型 | 描述 | 主要需求 |
|---------|------|---------|
| 中小企业 | 有融资需求的企业 | 快速获取信用报告、申请贷款 |
| 金融机构 | 银行、贷款公司 | 高效审批、风险管控 |
| 风控人员 | 内部风控团队 | 实时监控、自动预警 |
| 管理人员 | 平台运营方 | 数据分析、决策支持 |

## 3. MoSCoW 优先级定义

### 3.1 优先级分级标准

- **Must (必须完成)**: 核心业务功能，上线必备，阻塞性
- **Should (应该完成)**: 重要功能，提升体验，上线前完成
- **Could (可以完成)**: 增强功能，有时间再实现
- **Won't (不会完成)**: 本期不包含，待未来规划

### 3.2 功能优先级矩阵

```
| registered_capital | decimal | 注册资本 |
| establishment_date | date | 成立日期 |
| business_status | string | 经营状态 |
| industry | string | 所属行业 |
| credit_score | int | 信用评分(0-100) |
| risk_level | string | 风险等级 |
```

### 4.2 模块M4: 贷款申请流程

```
需求编号: M4
需求名称: 贷款产品申请与材料提交
优先级: MUST

功能描述:
- 用户选择贷款产品并发起申请

  created_at: datetime
  updated_at: datetime
}
```

### 4.3 模块M5: 自动化审批

```
需求编号: M5
需求名称: 贷款审批流程
优先级: MUST

审批流程:
[提交申请] → [自动预审] → [规则审核] → [人工复审] → [终审] → [结果通知]

规则引擎配置:
| 规则ID | 规则名称 | 条件 | 动作 |
|-------|---------|------|------|
| R001 | 企业状态检查 | business_status ≠ '存续' | 自动拒绝 |
| R002 | 信用分检查 | credit_score < 60 | 自动拒绝 |
| R003 | 额度限制 | loan_amount > 信用额度*1.5 | 标记待人工处理 |
| R004 | 行业限制 | industry in ['禁入行业'] | 自动拒绝 |
| R005 | 企业年龄 | establishment_date < 2年前 | 标记补充材料 |

审批状态流转:
draft → submitted → reviewing → [approved/rejected]
                      ↓
                needs_manual_review → manual_review → [approved/rejected]
```

### 4.4 模块M6: 风险评估

```
需求编号: M6
需求名称: 风险评估与评级系统
优先级: MUST

评分模型 (满分100分):

| 维度 | 权重 | 指标 |
|-----|------|------|
| 基础资质 | 30% | 成立年限、注册资本、经营状态 |
| 信用记录 | 35% | 信用评分、历史逾期、诉讼记录 |
| 财务状况 | 25% | 营收规模、资产负债、现金流 |
| 行业风险 | 10% | 行业周期、政策影响、竞争状况 |

风险等级定义:
| 等级 | 分数范围 | 说明 | 审批策略 |
|-----|---------|------|---------|
| AAA | 90-100 | 极低风险 | 自动通过 |
| AA | 80-89 | 低风险 | 自动通过 |
| A | 70-79 | 较低风险 | 自动通过 |
| BBB | 60-69 | 中等风险 | 人工复审 |
| BB | 50-59 | 较高风险 | 人工复审 |
| B | 40-49 | 高风险 | 人工终审 |
| C | 0-39 | 极高风险 | 自动拒绝 |

风控预警规则:
- 企业信用分连续下降>10分 → 预警
- 行业政策重大变化 → 预警
- 贷款申请数量异常增长 → 预警
```

## 11. 多租户与订阅计费架构（扩展设计）

### 11.1 多租户架构方案

#### 11.1.1 业务背景
支持多地市推广（焦作→河南全省→全国），实现省级-地市级-区县级多级架构，需满足：
- 省级平台统一管理、计费
- 地市级独立运营、数据隔离
- 区县级业务办理、数据归属

#### 11.1.2 多租户模式选择

| 隔离模式 | 适用场景 | 优点 | 缺点 | 推荐 |
|---------|---------|------|------|------|
| **独立库** | 省级/大型地市 | 完全隔离、故障不影响其他租户 | 成本高、运维复杂 | ✓ 地市级主库 |
| **共享库分Schema** | 中型地市 | 隔离较好、成本适中 | 迁移复杂 | 区县级 |
| **字段级隔离** | 小型区县/测试 | 成本最低 | 隔离性弱、查询性能差 | 初期/业务量小 |

**推荐策略：混合模式**
- **省级**: 独立数据库（核心数据、计费）
- **地市级**: 独立Schema或独立库（业务数据）
- **区县级**: 共享库分Schema（业务办理数据）

#### 11.1.3 数据隔离设计

```
┌─────────────────────────────────────────────────────────┐
│                    应用服务层                           │
│  (租户上下文 + 路由策略)                                │
├─────────────────────────────────────────────────────────┤
│  省级租户库    │  地市租户库      │  区县租户库         │
│  (河南)       │  (焦作/郑州...)   │  (各区县Schema)     │
├─────────────────────────────────────────────────────────┤
│                    统一数据服务层                       │
│  租户识别 → 路由 → 权限过滤                            │
└─────────────────────────────────────────────────────────┘
```

**租户标识体系：**
```json
{
  "tenant_id": "HENAN_PROVINCE",      // 省级租户ID
  "city_code": "410800",              // 地市代码（焦作）
  "district_code": "410802",          // 区县的代码（山阳区）
  "tenant_type": "PROVINCE|CITY|DISTRICT"
}
```

**核心表租户字段：**
| 表类型 | 租户字段 | 说明 |
|-------|---------|------|
| 企业数据 | tenant_id, city_code | 按地市隔离 |
| 贷款申请 | tenant_id, city_code, district_code | 归属到区县 |
| 用户 | tenant_id | 省级用户可管理全省 |
| 配置表 | tenant_id | 地市个性化配置 |

### 11.2 订阅计费模型

#### 11.2.1 套餐体系设计

| 套餐类型 | 适用对象 | 基础功能 | 进阶功能 | 定价策略 |
|---------|---------|---------|---------|---------|
| **基础版** | 试用/小型区县 | 企业查询、基础贷款申请 | - | 免费/99元/月 |
| **标准版** | 地市级 | 全功能贷款流程、基础统计 | API调用100次/天 | 999元/月 |
| **专业版** | 发达地市 | 高级风控、自定义审批规则 | API调用1000次/天 | 2999元/月 |
| **旗舰版** | 省级/大型 | AI审批、BI分析、专属客服 | 无限制 | 9999元/月 |

#### 11.2.2 计费维度

| 计费项 | 计费方式 | 说明 |
|-------|---------|------|
| **订阅套餐** | 月/年付 | 按租户级别收取固定费用 |
| **API调用量** | 阶梯计费 | 超出套餐额度后按次计费 |
| **贷款笔数** | 阶梯计费 | 月贷款申请>100笔后加收 |
| **增值服务** | 按需 | 高级风控模型、AI报告等 |

**计费公式：**
```
月度费用 = 套餐费 + max(0, API调用量 - 套餐额度) × 单价 + 超出笔数费用
```

#### 11.2.3 订阅管理流程

```
[选择套餐] → [配置租户] → [计费生效] → [周期扣费] → [续费/降级]
```

#### 11.4.3 数据同步策略

| 数据类型 | 同步方式 | 频率 | 说明 |
|---------|---------|------|------|
| 行政区划 | 全量同步 | 实时 | 省级下发 |
| 产品配置 | 全量/增量 | 实时 | 省级统一下发 |

- 跨地市查询 → 省级路由

#### 11.5.2 关键设计要点

1. **租户上下文传递**: 通过JWT/Header传递租户ID，全链路透传

3. **统一日志**: 租户ID入日志，便于问题排查和审计
4. **资源隔离**: 重要租户可独立部署服务实例
