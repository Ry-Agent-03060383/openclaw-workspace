## 二、架构设计

```
┌─────────────────────────────────────────┐
│           Controller 层 (控制层)          │
│  处理请求参数校验、响应封装、路由分发        │

│           Service 层 (业务层)            │
│  业务逻辑处理、事务管理                   │
├─────────────────────────────────────────┤

│           Entity 层 (实体层)             │
```

```
┌─────────────────────────────────────────┐
│           Views (视图层)                 │

```

## 三、核心功能模块设计

### 3.1 认证授权

- **JWT + Redis**: Token 存储在 Redis，支持失效处理
- **角色权限**: 企业、银行、第三方、运营四种角色
- **接口权限**: 基于注解的权限控制

```java
@PreAuthorize("hasRole('ENTERPRISE') or hasRole('BANK')")
public Result<LoanApplication> apply(LoanApplyDTO dto) { }
```

### 3.2 贷款流程

```
企业申请 → 智能匹配 → 预审 → 银行审核 → 放款 → 贷后监控 → 风险处置
```

### 3.3 AI 服务集成

- **对话服务**: 支持流式输出
- **上下文管理**: 会话历史存储
- **功能扩展**: 支持多种大模型切换

```java
// AI 对话服务示例
public Flux<String> chat(String message, String sessionId) {
    // 1. 获取历史上下文
    // 2. 调用大模型API
    // 3. 流式返回结果
}
```

### 3.4 贷后监控

- **定时任务**: 每日检查逾期情况
- **预警规则**: 自动触发预警
- **风险评分**: 动态计算企业风险

### 3.5 数据安全

- **敏感数据加密**: 密码、身份证号等
- **数据传输**: HTTPS 全站加密
- **日志审计**: 关键操作日志记录

## 四、开发规范建议

### 4.1 后端规范

- Controller 只做参数校验和响应封装
- Service 处理业务逻辑，标注事务
- Entity 使用 Lombok 简化代码
- DTO/VO 分离，输入输出独立
- 统一异常处理，定义业务异常

### 4.2 前端规范

- 使用 TypeScript 提高代码质量
- 组件按业务模块划分目录
- API 按模块封装，命名规范
- 使用 Pinia 管理状态
- CSS 使用 SCSS，预处理器

### 4.3 Git 分支管理

```
main (主分支)
├── develop (开发分支)
│   ├── feature/xxx (功能分支)
│   ├── fix/xxx (修复分支)
│   └── refactor/xxx (重构分支)
└── release/xxx (发布分支)
```

|------|------|------|
| 开发 | 开发调试 | dev.wisdom-finance.com |

### 5.2 容器化部署

```yaml
# docker-compose.yml (示例)
version: '3.8'
services:
  mysql:
    image: mysql:8.0
  redis:
    image: redis:7
  backend:
    build: ./backend
  frontend:
    build: ./frontend
    nginx: ...
```

### 5.3 建议技术组件

- **负载均衡**: Nginx
- **容器编排**: Docker Compose / Kubernetes
- **CI/CD**: Jenkins / GitLab CI
- **日志**: ELK (Elasticsearch + Logstash + Kibana)
- **监控**: Prometheus + Grafana
